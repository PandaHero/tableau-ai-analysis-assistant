# 洞察反思机制 - 需求文档

## 简介

本项目旨在为智能分析系统引入**任务级别的洞察反思机制**，借鉴 BettaFish 的反思设计，让 InsightAgent 能够自我评估生成的洞察质量，主动发现遗漏信息，并进行迭代改进。同时提供**用户可选的分析模式**（快速模式 vs 深度模式），平衡速度和质量。

### 核心目标

1. **自我评估** - InsightAgent 能够评估自己生成的洞察是否完整
2. **主动补充** - 发现遗漏信息时，主动进行补充分析
3. **迭代改进** - 通过反思循环，逐步提高洞察质量
4. **用户选择** - 提供快速模式和深度模式，让用户根据场景选择
5. **智能决策** - 系统自动评估任务复杂度，决定是否需要反思
6. **性能优化** - 通过选择性反思和轻量级反思，控制性能开销

### 参考项目

- **BettaFish** - 反思循环设计、ReflectionNode、ReflectionSummaryNode

## 术语表

### 核心概念
- **任务级反思（Task-Level Reflection）** - 针对单个任务的洞察进行自我评估和改进
- **分析级重规划（Analysis-Level Replanning）** - 综合多个任务的洞察，决定整体分析方向（已有功能）
- **自我评估（Self-Evaluation）** - Agent 评估自己生成的洞察质量
- **遗漏检测（Gap Detection）** - 识别洞察中缺失的关键信息
- **补充分析（Supplementary Analysis）** - 针对遗漏信息进行的额外分析
- **轻量级反思（Light Reflection）** - 基于现有数据的快速补充分析，不执行新查询
- **完整反思（Full Reflection）** - 可能执行补充查询的深度反思
- **复杂度评估（Complexity Assessment）** - 评估任务的复杂程度，决定反思策略

### 分析模式
- **快速模式（Fast Mode）** - 优先速度，最小化反思
- **深度模式（Deep Mode）** - 优先质量，充分反思
- **自动模式（Auto Mode）** - 系统根据任务复杂度自动决定

### 任务类型
- **简单任务（Simple Task）** - 数据明确、维度单一的任务
- **复杂任务（Complex Task）** - 数据量大、维度多、需要深度分析的任务
- **关键任务（Critical Task）** - 探索式分析中的聚焦任务

## 术语表（系统组件）

- **ReflectionEvaluator** - 反思评估器，评估洞察质量
- **ComplexityAssessor** - 复杂度评估器，评估任务复杂度
- **SupplementaryAnalyzer** - 补充分析器，执行补充分析
- **ReflectionStrategy** - 反思策略，决定反思方式

## 需求1: 洞察反思机制

**用户故事**: 作为 InsightAgent，我希望能够评估自己生成的洞察质量，发现遗漏信息，并进行补充分析，提高洞察的完整性和准确性

#### 验收标准

1. WHEN THE InsightAgent 生成初始洞察, THE System SHALL 评估洞察的完整性
2. WHEN THE System 发现遗漏信息, THE System SHALL 识别具体遗漏的方面
3. WHEN THE System 执行补充分析, THE System SHALL 基于遗漏方面生成补充洞察
4. WHEN THE System 合并洞察, THE System SHALL 整合初始洞察和补充洞察，去除冗余
5. WHEN THE System 完成反思循环, THE System SHALL 标记反思次数和改进点
6. THE System SHALL 支持最多2次反思循环
7. THE System SHALL 确保反思后的洞察质量 >= 初始洞察质量
8. THE System SHALL 确保反思不会引入错误信息

#### 反思流程

```
生成初始洞察
    ↓
反思评估器：评估完整性
    ↓
是否完整？
    ├─ 是 → 返回初始洞察
    └─ 否 → 识别遗漏方面
         ↓
    补充分析器：生成补充洞察
         ↓
    洞察合并器：合并洞察
         ↓
    反思次数 < 最大次数？
         ├─ 是 → 回到"反思评估器"
         └─ 否 → 返回最终洞察
```

#### 反思评估维度

**评估维度**：
1. **数据覆盖度** - 是否分析了所有关键数据点
2. **维度完整性** - 是否考虑了所有相关维度
3. **趋势分析** - 是否分析了时间趋势（如适用）
4. **异常识别** - 是否识别和解释了异常值
5. **对比分析** - 是否进行了必要的对比（如适用）
6. **根因挖掘** - 是否深入到根本原因（如适用）

**评估输出**：
```python
@dataclass
class ReflectionEvaluation:
    """反思评估结果"""
    is_complete: bool  # 是否完整
    completeness_score: float  # 完整性评分（0-1）
    missing_aspects: List[str]  # 遗漏的方面
    reasoning: str  # 评估理由
```

## 需求2: 用户可选的分析模式

**用户故事**: 作为业务数据分析师，我希望能够根据场景选择分析模式，在快速查询时选择快速模式，在深度分析时选择深度模式

#### 验收标准

1. WHEN THE User 发起分析, THE System SHALL 提供分析模式选择（快速/深度/自动）
2. WHEN THE User 选择快速模式, THE System SHALL 最小化反思，优先响应速度
3. WHEN THE User 选择深度模式, THE System SHALL 充分反思，优先分析质量
4. WHEN THE User 选择自动模式, THE System SHALL 根据任务复杂度自动决定
5. WHEN THE System 执行分析, THE System SHALL 在UI中显示当前使用的模式
6. THE System SHALL 记录用户的模式偏好，下次默认使用
7. THE System SHALL 在分析报告中说明使用的模式和反思次数

#### 分析模式定义

**快速模式（Fast Mode）**：
- 目标：快速响应，适合简单查询
- 策略：
  - 普通分析：不反思
  - 探索式分析：仅关键任务使用轻量级反思
- 预期时间：+10-20%
- 适用场景：日常数据查询、简单统计

**深度模式（Deep Mode）**：
- 目标：深度分析，适合复杂问题
- 策略：
  - 普通分析：复杂任务使用轻量级反思
  - 探索式分析：所有任务使用完整反思（最多2次）
- 预期时间：+50-75%
- 适用场景：根因分析、战略决策、重要报告

**自动模式（Auto Mode）**：
- 目标：平衡速度和质量
- 策略：根据任务复杂度自动决定
  - 简单任务：不反思
  - 中等任务：轻量级反思
  - 复杂任务：完整反思（1次）
- 预期时间：+30-40%
- 适用场景：默认模式，适合大多数场景

#### 模式切换

```python
@dataclass
class AnalysisRequest:
    """分析请求"""
    question: str
    mode: AnalysisMode  # FAST, DEEP, AUTO
    analysis_type: AnalysisType  # NORMAL, EXPLORATORY
```

## 需求3: 智能反思策略

**用户故事**: 作为系统，我需要根据任务特征和用户选择的模式，智能决定反思策略，避免不必要的反思开销

#### 验收标准

1. WHEN THE System 评估任务, THE System SHALL 计算任务复杂度评分（0-1）
2. WHEN THE System 决定反思策略, THE System SHALL 综合考虑任务复杂度和用户模式
3. WHEN THE System 执行轻量级反思, THE System SHALL 仅基于现有数据补充分析
4. WHEN THE System 执行完整反思, THE System SHALL 可以执行补充查询
5. THE System SHALL 记录每个任务的反思决策和理由
6. THE System SHALL 确保反思策略的一致性和可解释性

#### 反思策略矩阵

| 用户模式 | 任务复杂度 | 分析类型 | 反思策略 | 最大反思次数 |
|---------|-----------|---------|---------|------------|
| 快速模式 | 简单 | 普通 | 不反思 | 0 |
| 快速模式 | 简单 | 探索式 | 不反思 | 0 |
| 快速模式 | 中等 | 普通 | 不反思 | 0 |
| 快速模式 | 中等 | 探索式 | 轻量级反思 | 1 |
| 快速模式 | 复杂 | 普通 | 轻量级反思 | 1 |
| 快速模式 | 复杂 | 探索式 | 轻量级反思 | 1 |
| 快速模式 | 关键任务 | 探索式 | 完整反思 | 1 |

| 深度模式 | 简单 | 探索式 | 轻量级反思 | 1 |

| 深度模式 | 中等 | 探索式 | 完整反思 | 1 |

#### 轻量级反思 vs 完整反思

**轻量级反思**：
- 仅基于现有查询结果进行补充分析
- 不执行新的数据库查询
- 从不同角度重新分析现有数据
- 时间开销：+2-3秒

**完整反思**：
- 可以执行补充查询获取新数据
- 可以从不同维度或时间段查询
- 基于新数据生成补充洞察
- 时间开销：+5-8秒

## 需求4: 复杂度评估算法

**用户故事**: 作为系统，我需要准确评估任务的复杂度，为反思策略提供依据

#### 验收标准

1. WHEN THE System 评估任务复杂度, THE System SHALL 考虑多个维度
2. WHEN THE System 计算复杂度评分, THE System SHALL 输出0-1之间的评分
3. WHEN THE System 分类任务, THE System SHALL 根据评分分为简单/中等/复杂
4. THE System SHALL 确保复杂度评估的准确率 >= 85%
5. THE System SHALL 记录复杂度评估的详细信息

#### 复杂度评估维度

**数据维度（权重：30%）**：
- 数据行数：
  - < 100行：0分
  - 100-1000行：0.5分
  - > 1000行：1分
- 数据列数：
  - < 5列：0分
  - 5-10列：0.5分
  - > 10列：1分

**问题维度（权重：30%）**：
- 问题类型：
  - 简单查询（"是多少"）：0分
  - 对比分析（"哪个更高"）：0.5分
  - 根因分析（"为什么"）：1分
- 问题复杂度：
  - 单一指标：0分
  - 多指标对比：0.5分
  - 多维度交叉：1分

**分析维度（权重：20%）**：
- 维度数量：
  - 1个维度：0分
  - 2-3个维度：0.5分
  - > 3个维度：1分
- 时间跨度：
  - 单一时间点：0分
  - 短期（< 3个月）：0.5分
  - 长期（>= 3个月）：1分

**异常维度（权重：20%）**：
- 异常检测：
  - 无异常：0分
  - 有异常：1分
- 数据质量：
  - 无问题：0分
  - 有空值/异常值：0.5分

**计算公式**：
```
复杂度评分 = (数据维度 × 0.3) + (问题维度 × 0.3) + (分析维度 × 0.2) + (异常维度 × 0.2)
```

**分类标准**：
- 简单任务：评分 < 0.4
- 中等任务：评分 0.4-0.7
- 复杂任务：评分 > 0.7

## 需求5: 反思效果监控

**用户故事**: 作为系统管理员，我希望能够监控反思机制的效果，评估其对分析质量和性能的影响

#### 验收标准

1. WHEN THE System 完成反思, THE System SHALL 记录反思前后的洞察对比
2. WHEN THE System 生成报告, THE System SHALL 包含反思统计信息
3. WHEN THE System 监控性能, THE System SHALL 记录反思的时间开销
4. THE System SHALL 提供反思效果的可视化报表
5. THE System SHALL 支持按模式、任务类型统计反思效果

#### 监控指标

**质量指标**：
- 反思改进率：反思后洞察质量提升的百分比
- 遗漏发现率：成功发现遗漏信息的比例
- 错误纠正率：反思中纠正错误的比例
- 用户满意度：用户对反思后洞察的满意度

**性能指标**：
- 平均反思时间：每次反思的平均耗时
- 反思触发率：触发反思的任务比例
- 模式分布：各模式的使用比例
- 时间增加率：相比无反思的时间增加百分比

**业务指标**：
- 根因识别率：成功识别根因的比例
- 分析完整度：分析覆盖度的提升
- 用户采纳率：用户采纳反思建议的比例

## 技术约束

1. **性能约束**：
   - 快速模式下，总时间增加不超过20%
   - 深度模式下，总时间增加不超过75%
   - 单次反思时间不超过8秒

2. **质量约束**：
   - 反思不应引入错误信息
   - 反思后洞察质量应 >= 初始洞察
   - 复杂度评估准确率 >= 85%

3. **兼容性约束**：
   - 与现有的重规划机制兼容
   - 不影响现有的分段处理机制
   - 支持现有的所有分析类型

## 风险和缓解措施

### 风险1：反思增加响应时间
- **影响**：用户体验下降
- **缓解**：提供快速模式，默认使用自动模式
- **监控**：实时监控响应时间，超时则降级

### 风险2：反思引入错误信息
- **影响**：分析质量下降
- **缓解**：严格的评估机制，保守的补充策略
- **监控**：人工审核反思结果，建立反馈机制

### 风险3：反思与重规划冗余
- **影响**：重复分析，浪费资源
- **缓解**：任务级反思关注单个洞察，重规划关注整体方向
- **监控**：分析反思和重规划的重叠度
