## 技术选型

### 前端：Vue 3 + TypeScript + Vite

**选择Vue 3的理由**：
1. **渐进式框架**：更容易上手，模板语法直观，适合快速重构UI
2. **更好的TypeScript支持**：Vue 3原生支持TypeScript，类型推断更强
3. **组合式API**：与React Hooks类似但更灵活，逻辑复用更清晰
4. **更小的包体积**：运行时更轻量，适合嵌入式场景（Tableau Extension）
5. **更好的性能**：响应式系统基于Proxy，性能优于React的虚拟DOM diff
6. **官方生态完整**：Vue Router、Pinia、Vite官方维护，集成度高

**React的优势（但不适合本项目）**：
- 更大的社区和生态
- 更多的第三方组件库
- 但本项目需要完全自定义UI，不依赖现成组件库，Vue的灵活性更适合

**最终决策**：使用Vue 3 + TypeScript + Vite

### 后端：LangChain + LangGraph

**选择理由**：
1. **成熟的LLM编排框架**：LangChain提供丰富的工具和抽象
2. **图编排能力**：LangGraph支持复杂的多智能体工作流
3. **现有基础**：项目已使用LangChain和tableau_langchain，可直接扩展
4. **社区活跃**：持续更新，文档完善

## AI与代码的职责划分原则

本项目采用"AI做理解，代码做执行"的设计理念，明确区分AI和代码的职责边界。


### AI的职责（语义理解层）

AI负责所有需要语义理解、灵活决策和业务解读的任务：

1. **自然语言理解**：理解用户问题的业务意图、识别隐含需求、生成澄清问题
2. **字段理解与映射**：从完整数据模型中选择正确字段、理解字段关系和层级
3. **问题拆分与结构化**：将复杂问题拆分为子问题、识别依赖关系、评估复杂度
4. **智能决策与推荐**：决定是否重规划、生成后续问题推荐、选择重规划类型
5. **异常解读与洞察**：为统计异常提供业务解释、提取关键发现、生成业务洞察
6. **语义补全**：补全模糊的时间表达、筛选条件、聚合方式等

**AI的输出**：结构化的规格（Spec）、决策结果、可视化图表、分析报告，而非可执行代码或精确数据

### 代码的职责（确定性执行层）

代码负责所有需要精确性、稳定性和可预测性的任务：

1. **精确的查询生成**：基于AI的规格生成标准VizQL JSON，使用代码模板而非AI
2. **数据处理与计算**：数据合并、去重、聚合、排序等，使用Pandas/SQL等工具
3. **系统流程控制**：任务调度、并发控制、超时处理、重试机制，使用LangGraph编排
4. **统计检测**：异常检测（Z-score、IQR）、数据质量检查，使用统计方法
5. **验证与校验**：验证字段存在性、检查依赖关系、验证JSON结构
6. **性能优化**：缓存管理、查询合并、数据量估算、资源控制

**代码的输出**：可执行的查询语句、处理后的数据、客观的统计报告

### 协作模式

```
用户问题
  ↓
[母Agent - AI理解] → StructuredQuestionSpec（结构化规格）
  ↓
[QueryBuilder - 代码生成] → VizQL查询JSON
  ↓
[VDS Client - 代码执行] → 查询结果
  ↓
[Statistics - 代码统计] → 统计报告（客观数据）
  ↓
[分析Agent - AI解读] → 业务洞察（AnalysisResult）
  ↓
[母Agent - AI决策] → 重规划建议
  ↓
[代码验证] → [母Agent - AI生成] → 新一轮规格 → ...
```

### 真正的"Agent"有三个（基于职责清晰划分）

1. **规划Agent（Planner Agent）**：问题理解、任务拆分、重规划决策
2. **分析Agent（Analyzer Agent）**：单个子任务的业务洞察生成
3. **总结Agent（Summarizer Agent）**：整合所有结果、生成最终报告

**为什么是三个Agent？**
- **职责清晰**：每个Agent只做一件事（规划、分析、总结）
- **易于优化**：可以独立优化每个Agent的提示词和策略
- **符合单一职责原则**：避免一个Agent职责过多过杂

其他都是**纯代码组件**（详见下文"核心组件"章节）：
- QueryBuilder（查询构建器）：生成VizQL查询JSON
- VDS Executor（VDS执行器）：调用VDS API获取数据
- Statistics Detector（统计检测器）：客观的统计分析
- Data Merger（数据合并器）：智能合并多个查询结果
- Task Scheduler（任务调度器）：并行执行、超时控制、失败处理

### 关键原则

1. **AI做"理解"，代码做"执行"**：AI负责语义理解和决策，代码负责精确执行
2. **AI做"生成规格"，代码做"生成代码"**：AI生成意图描述，代码生成查询语句
3. **AI做"分析"，代码做"检测"**：AI分析业务含义，代码执行客观检测
4. **AI做"推荐"，代码做"验证"**：AI推荐方向，代码验证可行性
5. **AI做"灵活决策"，代码做"确定性控制"**：AI处理不确定性，代码保证稳定性

## VizQL查询能力说明

本节说明VizQL Data Service的查询能力，帮助AI在问题拆分时做出正确决策。

详细内容参见：[VizQL查询能力详解](./appendix/vizql-capabilities.md)

### 核心能力概览

| 能力 | 说明 | 示例 |
|------|------|------|
| **多维度** | 一个查询可以包含多个维度字段 | [地区, 产品类别, 门店] |
| **多度量** | 一个查询可以包含多个度量字段 | [销售额, 利润, 订单量] |
| **多筛选** | 一个查询可以包含多个筛选条件 | [日期范围, 地区=北京, 销售额>1000] |
| **聚合函数** | 支持SUM、AVG、COUNT、MIN、MAX等 | SUM(销售额)、AVG(利润率) |
| **日期函数** | 支持YEAR、MONTH、DAY、TRUNC_YEAR等 | YEAR(订单日期)、MONTH(订单日期) |
| **计算字段** | 支持自定义计算表达式 | 利润率 = 利润 / 销售额 |
| **排序** | 支持多字段排序，指定优先级 | sortPriority: 1, 2, 3... |
| **TopN** | 支持TopNFilter限制结果数量 | Top 10门店 |

### 问题拆分决策指南

**不需要拆分的情况**（一个VizQL查询即可）：
- ✅ 多个度量 + 相同维度组合 + 相同时间段
  - 例："2016年各门店的销售额、利润和订单量"
- ✅ 多个维度 + 多个度量
  - 例："各地区各产品类别的销售额和利润"
- ✅ 多个筛选条件 + 单个时间段
  - 例："2016年北京和上海地区销售额超过1000的门店"

**需要拆分的情况**（需要多个VizQL查询）：
- ❌ 多个时间段对比（同比、环比）
  - 例："2016年和2015年各门店的销售额对比" → 拆分为2个查询
- ❌ 不同维度组合（先总体后明细）
  - 例："先看总体销售额，再看各地区明细" → 拆分为2个查询
- ❌ 计算依赖（需要先计算A再计算B）
  - 例："计算各门店销售额占比" → 先查总销售额，再查各门店销售额
- ❌ 不同筛选条件组合对比
  - 例："对比高价产品和低价产品的销售情况" → 拆分为2个查询

### 核心组件

#### 1. 规划Agent (Planner Agent)
- **职责**：问题理解、任务拆分、重规划决策
- **输入**：用户问题、数据源元数据、历史上下文
- **输出**：结构化子任务列表（StructuredQuestionSpec）、重规划建议
- **技术**：LangChain + LLM
- **核心能力**：
  - 问题理解（复杂度评估、类型识别、字段选择）
  - 任务拆分（基于VizQL能力决策、依赖关系分析）
  - 重规划决策（评估是否需要重规划、生成推荐问题）
- **说明**：专注于"规划"，不负责数据分析和总结

#### 2. 分析Agent (Analyzer Agent)
- **职责**：对单个子任务的统计报告进行业务解读
- **输入**：统计报告（StatisticalReport，来自Statistics Detector）
- **输出**：业务洞察（AnalysisResult）
- **技术**：LangChain + LLM
- **核心能力**：
  - 异常解读（为统计异常提供业务解释）
  - 趋势洞察（解释趋势的业务含义）
  - 行动建议（提供具体可执行的建议）
- **说明**：专注于"单个任务的业务洞察"，不负责整体总结

#### 3. 总结Agent (Summarizer Agent)
- **职责**：整合所有轮次的分析结果，生成最终报告
- **输入**：所有轮次的分析结果、合并后的数据、重规划历史
- **输出**：最终分析报告（FinalSummary）
- **技术**：LangChain + LLM
- **核心能力**：
  - 结果整合（去重、排序、分类关键发现）
  - 执行摘要生成（一句话回答原始问题）
  - 分析路径回顾（展示分析思路）
  - 后续探索建议（推荐深入分析方向）
- **说明**：专注于"最终总结"，不负责问题拆分和单个任务分析

### 纯代码组件（不涉及LLM）

#### 3. VizQL查询构建器 (Query Builder)
- **职责**：根据StructuredQuestionSpec生成符合VDS规范的JSON查询
- **输入**：StructuredQuestionSpec（包含dims、metrics、filters等）
- **输出**：VizQL查询JSON
- **技术**：Builder模式 + 代码模板
- **核心类**：
  - BasicQueryBuilder：基础查询
  - TimeSeriesQueryBuilder：时间序列查询
  - RankingQueryBuilder：排名查询
  - ComparisonQueryBuilder：对比查询
- **说明**：使用纯代码模板生成查询，确保100%正确，避免AI幻觉

#### 4. VDS查询执行器 (VDS Query Executor)
- **职责**：执行VizQL查询，获取数据
- **输入**：VizQL查询JSON
- **输出**：查询结果（DataFrame）
- **技术**：VDS API Client + 重试机制
- **功能**：
  - 调用Tableau VDS API
  - 处理分页（自动获取所有页）
  - 错误处理和重试（指数退避）
  - 超时控制
- **说明**：纯API调用，不涉及任何AI逻辑

#### 5. 统计检测器 (Statistics Detector)
- **职责**：对查询结果进行客观的统计分析
- **输入**：查询结果（DataFrame）
- **输出**：统计报告（StatisticalReport）
- **技术**：Pandas + NumPy + SciPy
- **功能**：
  - 描述性统计（均值、中位数、标准差等）
  - 异常检测（Z-score、IQR、MAD、孤立森林）
  - 趋势分析（线性回归、Mann-Kendall检验）
  - 数据质量检查（完整性、一致性、准确性）
- **说明**：纯统计方法，不涉及AI，输出客观数据

#### 6. 数据合并器 (Data Merger)
- **职责**：智能合并多个子任务的查询结果
- **输入**：多个查询结果（List[DataFrame]）
- **输出**：合并后的数据（DataFrame）+ 数据质量评分
- **技术**：Pandas + 代码规则
- **功能**：
  - **Union合并**：纵向拼接（相同结构的数据）
  - **Join合并**：横向拼接（按共同维度关联）
  - **Append合并**：简单追加（不同结构的数据）
  - **数据对齐**：处理缺失值、类型转换、列名标准化
  - **去重处理**：基于主键或全行去重
  - **质量检查**：检测数据冲突、不一致、异常值
- **合并策略选择**：
  - 基于子任务的dims和metrics自动选择合并策略
  - 如果dims相同 → Union合并
  - 如果dims部分重叠 → Join合并
  - 如果dims完全不同 → Append合并
- **说明**：使用确定性的代码规则，不涉及AI决策

#### 7. 任务调度器 (Task Scheduler)
- **职责**：在execution_node内部，并行执行同一stage的多个子任务
- **输入**：子任务列表（List[StructuredQuestionSpec]）
- **输出**：执行结果列表（List[ExecutionResult]）
- **技术**：ThreadPoolExecutor + 状态机
- **功能**：
  - 按stage分组执行（保证依赖关系）
  - 同stage内并行执行（最多3个并发）
  - 动态超时控制（基于数据量和复杂度）
  - 智能重试（指数退避，最多2次）
  - 失败处理和降级策略
  - 实时进度反馈（SSE推送）
  - 资源监控（内存、CPU使用率）
- **说明**：
  - LangGraph负责高层工作流编排（节点间的流转）
  - TaskScheduler负责execution_node内部的子任务并行执行
  - 原因：子任务数量动态变化，需要细粒度的超时、重试、资源控制

## 任务调度的两层架构设计

### 为什么需要两层？

本项目采用**LangGraph + ThreadPoolExecutor**的两层任务调度架构，而不是完全依赖LangGraph的并发能力。

### 第一层：LangGraph工作流编排（高层编排）

**职责**：编排整体分析流程的节点流转

```
metadata_node → planning_node → execution_node → merge_node
    → replan_decision_node → replan_interaction_node
    → replan_generation_node → execution_node (新一轮) → ...
```

**并发粒度**：节点级别
- 不同节点之间是串行的（有依赖关系）
- 但可以并行执行多个独立的分支（如果有）

**技术**：LangGraph StateGraph

**适用场景**：
- ✅ 工作流结构相对固定
- ✅ 节点数量有限（10个左右）
- ✅ 需要状态管理和检查点机制

### 第二层：ThreadPoolExecutor任务并行（execution_node内部）

**职责**：在execution_node内部，并行执行同一stage的多个子任务

```python
def execution_node(state: WorkflowState):
    subtasks = state["subtasks"]

    # 按stage分组
    stages = group_by_stage(subtasks)

    # 依次执行每个stage
    for stage_num, stage_tasks in stages.items():
        # 使用ThreadPoolExecutor并行执行同stage任务
        with ThreadPoolExecutor(max_workers=3) as executor:
            futures = {executor.submit(execute_task, task): task
                      for task in stage_tasks}

            # 收集结果，处理失败
            for future in as_completed(futures):
                task = futures[future]
                try:
                    result = future.result(timeout=task.timeout)
                    results.append(result)
                except Exception as e:
                    handle_task_failure(task, e)

    return {...state, "execution_results": results}
```

**并发粒度**：子任务级别
- 同一stage内的任务并行执行
- 不同stage之间串行执行（保证依赖关系）

**技术**：ThreadPoolExecutor + 超时控制 + 重试机制

**适用场景**：
- ✅ 任务数量动态变化（基于AI的问题拆分，无法预知）
- ✅ 需要细粒度控制（超时、重试、降级、资源监控）
- ✅ 需要实时进度反馈（每个任务的执行状态）
- ✅ 需要灵活的并发数控制（基于系统资源动态调整）

### 为什么不全部用LangGraph？

#### 方案对比

| 方案 | 优点 | 缺点 | 适用性 |
|------|------|------|--------|
| **全部用LangGraph** | 统一的编排框架、原生并发支持、可视化清晰 | 动态节点添加复杂、子任务过多时图庞大、细粒度控制困难 | ❌ 不适合 |
| **LangGraph + ThreadPoolExecutor**（当前） | 高层清晰、低层灵活、易于控制并发和资源 | 两层架构需要理解 | ✅ 推荐 |
| **全部用ThreadPoolExecutor** | 实现简单、控制灵活 | 缺少状态管理、检查点、可视化 | ❌ 不适合 |

#### 具体原因

1. **子任务数量动态变化**
   - AI拆分的子任务数量不固定（可能3个，也可能10个）
   - LangGraph需要预先定义节点，动态添加节点较复杂

2. **需要细粒度控制**
   - 每个子任务需要独立的超时时间（基于数据量和复杂度）
   - 需要智能重试（指数退避、降级策略）
   - 需要资源监控（内存、CPU、数据库连接）
   - ThreadPoolExecutor提供更灵活的控制

3. **实时进度反馈**
   - 需要实时推送每个子任务的执行状态（开始、进行中、完成、失败）
   - ThreadPoolExecutor + as_completed()更容易实现

4. **并发数动态调整**
   - 需要根据系统资源使用率动态调整并发数（3 → 2 → 1）
   - ThreadPoolExecutor的max_workers可以动态调整

### 架构图

```
│              LangGraph工作流（第一层）                    │
│                                                           │
│  metadata → planning → execution → merge → replan        │

│                           ↓                               │

```

### 总结

- **LangGraph**：负责高层工作流编排，提供状态管理、检查点、可视化
- **ThreadPoolExecutor**：负责子任务并行执行，提供细粒度控制、超时、重试
- **两者结合**：既有清晰的工作流结构，又有灵活的任务调度能力
