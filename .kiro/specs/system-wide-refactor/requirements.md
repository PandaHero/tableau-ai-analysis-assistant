# 需求文档

## 简介

本文档规定了 Analytics Assistant 系统级重构的需求。重构旨在最大化代码复用、提升模块化程度、优化语义理解能力，并建立可扩展的架构以支持未来扩展。

该系统是基于 LangChain/LangGraph 的 AI 编排平台，通过多个专业化 Agent 实现自然语言数据分析。重构将完全重组代码库以提升可维护性、降低 token 消耗、增强准确性。本次重构不要求向后兼容，可以进行破坏性变更以实现最优架构。

**项目定位**：通用 BI 分析助手平台
- 目前实现：Tableau 平台支持
- 未来扩展：Power BI、Looker 等其他 BI 平台

## 术语表

- **系统（System）**: Analytics Assistant 平台（通用 BI 分析助手）
- **Agent**: 执行特定分析任务的专业化 LangGraph 节点（如 SemanticParser、FieldMapper）
- **组件（Component）**: Agent 内部的可复用功能模块（如 Step1Component、PreprocessComponent）
- **中间件（Middleware）**: 包装 Agent 执行的横切关注点处理器（如 OutputValidationMiddleware）
- **RAG**: 检索增强生成基础设施，用于字段映射和维度层级推断
- **语义解析器（Semantic_Parser）**: 负责理解用户意图并将自然语言转换为语义查询的核心 Agent
- **LangGraph**: 用于构建有状态多 Agent 工作流的编排框架
- **LangChain**: LLM 集成和工具调用的基础库
- **模型管理器（ModelManager）**: 管理多个 LLM 和 Embedding 模型配置的集中式服务
- **Token 消耗（Token_Consumption）**: 发送到 LLM API 的 token 数量，直接影响成本和延迟
- **基于属性的测试（Property-Based_Testing）**: 验证生成输入的通用属性的测试方法

## 需求

### 需求 1: 架构现代化

**用户故事：** 作为系统架构师，我希望有一个模块化的分层架构，具有清晰的关注点分离，以便系统易于维护、测试和扩展。

#### 验收标准

1. 系统应将代码组织为五个不同的层次：核心层（Core）、平台层（Platform）、Agent 层、编排层（Orchestration）和基础设施层（Infrastructure）
2. 当一个层需要另一个层的功能时，系统应仅依赖于较低层（Core ← Platform ← Agent ← Orchestration，Infrastructure 为横向层）
3. 系统应使用抽象基类定义层之间的显式接口
4. 添加新功能时，系统应根据其职责将代码放置在适当的层中
5. 系统应在架构文档中记录层职责和依赖规则

### 需求 2: 组件复用最大化

**用户故事：** 作为开发人员，我希望在所有 Agent 中复用现有的中间件和基础设施组件，以避免代码重复并保持一致性。

#### 验收标准

1. 系统应将常见的 Agent 模式提取为可复用的基础组件
2. 当多个 Agent 需要类似功能时，系统应在 agents/base 模块中提供共享组件
3. 系统应在所有 Agent 中复用现有中间件（FilesystemMiddleware 等 LangChain 自带中间件）
4. 系统应为所有检索需求复用统一的 RAG 基础设施（infra/rag）
5. 系统应记录可复用组件的使用示例和集成模式

**注**：部分中间件（PatchToolCallsMiddleware、OutputValidationMiddleware）将在重构后评估是否仍需保留

### 需求 3: 语义理解优化

**用户故事：** 作为产品负责人，我希望语义理解模块以更低的 token 消耗实现更高的准确性，以便系统以更低的成本提供更好的结果。

#### 验收标准

1. 处理用户查询时，系统应在 LLM 调用之前应用基于规则的预处理以减少 token 使用
2. 系统应实现三层意图路由策略（L0 规则、L1 小模型、L2 兜底）以最小化不必要的 LLM 调用
3. 执行 Schema Linking 时，系统应使用混合检索（向量 + 关键词 + 精确匹配）将候选字段集从 O(n) 减少到 O(k)
4. 系统应缓存预处理结果（规范化问题、提取的术语、时间上下文）以避免冗余计算
5. 生成提示时，系统应仅包含相关的 Schema 候选项而不是完整的数据模型
6. 系统应实现提示压缩技术，相比基线减少至少 30% 的 token 数量
7. 重试失败操作时，系统应提供结构化错误反馈以指导 LLM 纠正，而无需重复完整上下文

### 需求 4: 模块化 Agent 设计

**用户故事：** 作为开发人员，我希望 Agent 由独立的、可测试的组件组成，具有清晰的接口，以便我可以修改和测试单个部分而不影响整个系统。

#### 验收标准

1. 系统应将每个 Agent 分解为离散组件（如 IntentRouter、PreprocessComponent、Step1Component）
2. 组件执行时，系统应接受明确定义的输入参数并返回结构化输出模型
3. 系统应使用 Pydantic 模型定义组件接口的输入和输出
4. 测试组件时，系统应支持在隔离环境中进行单元测试，无需完整的 Agent 上下文
5. 系统应记录组件的职责、输入、输出和依赖关系
6. 组件失败时，系统应提供结构化错误信息以支持针对性的重试策略

### 需求 5: 接口规范与清晰性

**用户故事：** 作为系统架构师，我希望模块之间有明确定义的接口，以便我可以独立演进组件并保持代码清晰。

#### 验收标准

1. 系统应使用抽象基类或协议定义所有模块间接口
2. 系统应记录接口契约，包括输入/输出模式、错误条件和行为保证
3. 系统应通过集成测试验证接口合规性
4. 接口变更时，系统应更新所有依赖该接口的代码
5. 系统应提供清晰的接口文档和使用示例
6. 系统应确保接口设计符合单一职责原则

### 需求 6: 分阶段实施策略

**用户故事：** 作为项目经理，我希望有一个增量交付价值的分阶段实施计划，以便我们可以在继续之前验证每个阶段的正确性。

#### 验收标准

1. 系统应在 5-7 个不同阶段实施重构，每个阶段有明确的交付物
2. 完成每个阶段时，系统应确保该阶段的功能完整性和正确性
3. 系统应为每个阶段提供回滚机制以应对关键问题
4. 系统应在进入下一阶段之前通过全面测试验证每个阶段
5. 系统应为每个阶段记录实施指南，包括架构变更和数据迁移步骤
6. 每个阶段应有独立的验收标准和质量门禁

### 需求 7: 框架选型与论证

**用户故事：** 作为技术负责人，我希望有详细的框架选项对比和清晰的论证，以便我可以就技术选择做出明智的决策。

#### 验收标准

1. 系统应评估 LangChain/LangGraph 与替代编排框架的对比
2. 比较框架时，系统应评估以下标准：生态系统成熟度、流式支持、状态管理、中间件能力和社区支持
3. 系统应记录继续使用 LangChain/LangGraph 的理由
4. 系统应识别框架限制和缓解策略
5. 系统应评估替代 RAG 框架（LlamaIndex、Haystack）并论证当前方法
6. 系统应记录模型管理方法并论证 ModelManager 设计

### 需求 8: 风险评估与缓解

**用户故事：** 作为项目经理，我希望有全面的风险评估和缓解策略，以便我可以主动预测和解决潜在问题。

#### 验收标准

1. 系统应识别技术风险，包括：破坏性变更、性能下降、数据迁移问题和集成失败
2. 识别风险时，系统应评估可能性（低/中/高）和影响（低/中/高）
3. 系统应为每个识别的风险提供具体的缓解策略
4. 系统应为关键失败场景定义回滚程序
5. 系统应为每个阶段建立成功标准和监控指标
6. 系统应为高影响风险记录应急计划

### 需求 9: 语义解析器深度优化

**用户故事：** 作为数据科学家，我希望语义解析器利用先进的 NLP 技术和高效的提示工程，以便在最小化 API 成本的同时准确理解复杂查询。

#### 验收标准

1. 处理查询时，系统应在 LLM 调用之前应用文本规范化（全角转半角、空白整合、单位标准化）
2. 系统应使用基于规则的解析器提取时间表达式，以避免 LLM 在日期上的幻觉
3. 系统应使用 NLP 技术（NER、依存句法分析）从查询中提取业务术语以指导 Schema Linking
4. 系统应构建规范化问题，在改写中保持稳定以提高缓存命中率
5. 执行 Schema Linking 时，系统应结合精确匹配、N-gram 模糊匹配和向量相似度，采用两阶段分数融合
6. 系统应实现批量 Embedding 优化以减少向量检索的 API 调用
7. 生成 Step1 提示时，系统应仅包含 top-k Schema 候选项（k≤20）而不是完整 Schema
8. 系统应向 ReAct 错误处理器提供结构化错误反馈，以实现针对性纠正而无需重复完整上下文

### 需求 10: 测试策略与质量保证

**用户故事：** 作为 QA 工程师，我希望有全面的测试覆盖，包括单元测试、集成测试和基于属性的测试，以便我可以确保系统正确性并及早发现回归。

#### 验收标准

1. 系统应为核心业务逻辑实现至少 80% 的代码覆盖率
2. 测试组件时，系统应为隔离的单个组件编写单元测试
3. 系统应为 Agent 工作流编写端到端集成测试
4. 系统应为关键正确性属性实现基于属性的测试（如往返序列化、不变量保持）
5. 运行基于属性的测试时，系统应为每个属性执行至少 100 次迭代
6. 系统应维护在 5 分钟内运行的测试套件以获得快速反馈
7. 系统应记录测试模式并为常见场景提供测试工具

### 需求 11: 性能优化

**用户故事：** 作为系统管理员，我希望系统最小化延迟和资源消耗，以便用户获得快速响应且运营成本保持可控。

#### 验收标准

1. 处理查询时，系统应在 3 秒内完成 90% 请求的语义解析
2. 系统应缓存频繁访问的数据（数据模型、维度层级、字段索引）以减少冗余计算
3. 系统应为 BI 平台 API 调用实现连接池（如 Tableau VizQL API）
4. 执行 RAG 检索时，系统应使用批量 Embedding API 以减少网络开销
5. 系统应实现流式响应以向用户提供增量反馈
6. 系统应监控和记录性能指标（延迟、token 数量、缓存命中率）以持续优化
7. 检测到性能下降时，系统应提供警报和诊断信息

### 需求 12: 可观测性与监控

**用户故事：** 作为 DevOps 工程师，我希望有全面的日志、指标和追踪，以便我可以快速诊断问题并监控系统健康状况。

#### 验收标准

1. 系统应记录所有 LLM 调用的请求/响应详情、token 数量和延迟
2. 发生错误时，系统应记录结构化错误信息，包括堆栈跟踪、上下文和恢复操作
3. 系统应公开指标，包括：请求数量、成功率、平均延迟、token 消耗、缓存命中率
4. 系统应实现分布式追踪以跨 Agent 边界跟踪请求
5. 发生性能异常时，系统应生成带有诊断上下文的警报
6. 系统应提供仪表板以实时监控关键指标
7. 系统应保留日志至少 30 天以进行历史分析

### 需求 13: 配置管理

**用户故事：** 作为系统管理员，我希望有集中的、环境特定的配置管理，以便我可以在不同环境中部署系统而无需更改代码。

#### 验收标准

1. 系统应支持多个配置源（环境变量、配置文件、运行时覆盖）
2. 加载配置时，系统应验证所有必需设置并为缺失值提供清晰的错误消息
3. 系统应支持环境特定配置（开发、预发布、生产）
4. 系统应允许在不重启的情况下对非关键设置进行运行时配置更新
5. 系统应记录所有配置选项，包括描述、默认值和有效范围
6. 系统应加密静态敏感配置值（API 密钥、密码）
7. 配置变更时，系统应记录变更及时间戳和用户信息

### 需求 14: 文档与知识转移

**用户故事：** 作为新开发人员，我希望有涵盖架构、设计模式和实施指南的全面文档，以便我可以快速理解系统并做出贡献。

#### 验收标准

1. 系统应提供涵盖层职责、数据流和关键设计决策的架构文档
2. 系统应记录所有公共 API，包括描述、参数、返回值和使用示例
3. 系统应为常见任务提供实施指南（添加新 Agent、扩展组件、集成中间件）
4. 系统应记录代码库中使用的设计模式及其理由和示例
5. 系统应维护变更日志，记录所有重大变更、弃用和迁移
6. 系统应为常见问题提供故障排除指南，包括诊断步骤和解决方案
7. 系统应包含关键工作流的代码示例和教程

### 需求 15: 部署与回滚

**用户故事：** 作为系统管理员，我希望有清晰的部署策略和回滚方案，以便安全地部署新架构。

**重要说明**：项目未上线，不存在历史数据，因此不需要数据迁移工具。

#### 验收标准

1. 系统应记录所有架构变更和数据模型变更
2. 系统应提供部署检查清单以验证部署完整性
3. 系统应提供回滚方案以应对部署失败场景（Git Tag + 配置回滚）
4. 系统应记录新旧系统的功能差异和改进点
5. 系统应支持分阶段部署（7 个独立阶段，每个阶段可独立回滚）
6. 系统应为每个阶段提供验证标准和测试用例
7. 系统应提供环境配置管理（开发、预发布、生产）
