## 1. 概述

### 1.1 背景

当前 Tableau AI Analysis Assistant 的 Agent 架构存在以下核心问题：
- LLM 调用次数过多（5-13 次/查询）
- Token 消耗高（传递完整数据模型）
- 中间件冗余（8 个中间件，部分功能重复）

### 1.2 优化目标

| 指标 | 当前 | 目标 | 改进 |
|------|------|------|------|
| LLM 调用次数 | 5-13 次 | N+2 次 | ~50% |
| Token 消耗 | 高 | 低 | ~70% |
| 中间件数量 | 8 个 | 6 个 | 25% |
| 响应时间 | 24-46s | 14-24s | ~40% |

### 1.3 设计原则

1. **LLM 做决策，工具做执行**
2. **减少 LLM 记忆负担**（不传递完整数据模型）
3. **利用 LLM 的选择能力**（从候选中选择，而非生成）
4. **结构化输出保证格式**（JSON 模式 + Pydantic 验证）
5. **使用 LangChain 内置中间件**（不重复造轮子）

## 3. 模块设计

### 3.1 UnifiedSemanticParser

**职责**：将用户自然语言问题转换为结构化查询

**优化**：合并原 Step1 + Step2 为单次 LLM 调用

```
┌─────────────────────────────────────────────────────────────────┐
│                  UnifiedSemanticParser                           │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  输入：用户问题 + 对话历史                                       │
│                                                                 │
│  处理流程：                                                      │
│  1. 语义理解 (What/Where/How)                                   │
│  2. 意图分类 (DATA_QUERY/CLARIFICATION/GENERAL)                 │
│  3. 计算推理 (是否需要 LOD/TABLE_CALC)                          │
│  4. 自我验证                                                    │
│                                                                 │
│  输出：UnifiedSemanticOutput                                    │
│  - intent: 意图类型                                             │
│  - dimensions: 维度列表                                         │
│  - measures: 度量列表                                           │
│  - filters: 筛选条件                                            │
│  - computation: 复杂计算（可选）                                │
│  - reasoning: 推理过程                                          │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

**职责**：链式分析数据，生成洞察报告


### 3.4 Observer 错误处理

**职责**：检查 Pipeline 执行结果，决定下一步操作

**决策类型**：

| 决策 | 说明 | 触发条件 |
|------|------|---------|
| CONTINUE | 执行成功，继续下一步 | 工具执行成功，结果有效 |
| RETRY | 可恢复错误，重试指定步骤 | 字段映射失败、查询语法错误、超时 |
| CLARIFY | 需要用户澄清 | 字段无法映射、结果为空、歧义 |
| ABORT | 不可恢复错误，终止执行 | 权限错误、数据源不存在、系统错误 |

**错误分类与处理**：

| 错误类型 | 决策 | 处理方式 |
|---------|------|---------|
| 字段映射失败（RAG 无结果） | RETRY → map_fields | 使用 LLM Fallback |
| 字段映射失败（LLM 也无法映射） | CLARIFY | 询问用户字段含义 |
| 查询构建失败（语法错误） | RETRY → build_query | 修正查询语法 |
| 查询执行失败（权限错误） | ABORT | 返回权限错误消息 |
| 查询执行失败（超时） | RETRY → execute | 简化查询重试 |
| 查询执行失败（数据源不存在） | ABORT | 返回数据源错误 |
| 结果为空 | CLARIFY | 询问用户是否调整条件 |
| 网络错误 | RETRY | 由 ToolRetryMiddleware 处理 |

**注意**：Observer 不是 ReAct 模式。Observer 是错误检查组件，只在工具执行后检查结果并决策。Replanner 是可选的反思组件，用于评估分析完整性。

### 3.5 中间件架构

**优化**：从 8 个中间件简化为 6 个

| 中间件 | 来源 | 职责 | 状态 |
|--------|------|------|------|
| TodoListMiddleware | LangChain | 任务队列管理 | ✅ 保留 |
| SummarizationMiddleware | LangChain | 对话历史压缩 | ✅ 保留 |
| ModelRetryMiddleware | LangChain | LLM 调用重试 | ✅ 保留 |
| ToolRetryMiddleware | LangChain | 工具调用重试 | ✅ 保留 |
| FilesystemMiddleware | 自定义 | 大文件缓存 | ✅ 保留 |
| HumanInTheLoopMiddleware | LangChain | 人工确认 | ✅ 保留（可选）|
| OutputValidationMiddleware | 自定义 | 输出验证 | ❌ 移除 |
| PatchToolCallsMiddleware | 自定义 | 修复悬空工具调用 | ❌ 移除 |

**移除原因**：
- `OutputValidationMiddleware`：JSON 模式 + Pydantic 验证已处理
- `PatchToolCallsMiddleware`：JSON 模式不会产生悬空工具调用

### 6.1 核心 Schema

```python
class Intent(str, Enum):
    """意图类型"""
    DATA_QUERY = "DATA_QUERY"
    CLARIFICATION = "CLARIFICATION"
    GENERAL = "GENERAL"


class CalcType(str, Enum):
    """计算类型（精简为 5 种）"""
    RANK = "RANK"
    PERCENT = "PERCENT"
    RUNNING = "RUNNING"
    DIFF = "DIFF"
    NONE = "NONE"


class Dimension(BaseModel):
    """维度字段"""
    field: str = Field(description="字段名")
    granularity: Optional[Literal["YEAR", "MONTH", "DAY"]] = Field(
        default=None, description="日期粒度"
    )


class Measure(BaseModel):
    """度量字段"""
    field: str = Field(description="字段名")
    aggregation: str = Field(default="SUM", description="聚合函数")


class Filter(BaseModel):
    """筛选条件"""
    field: str = Field(description="字段名")
    operator: str = Field(description="操作符")
    value: str | list = Field(description="筛选值")


class Computation(BaseModel):
    """复杂计算"""
    calc_type: CalcType = Field(description="计算类型")
    target: str = Field(description="目标度量字段")
    partition_by: List[str] = Field(default_factory=list, description="分区字段")


class UnifiedSemanticOutput(BaseModel):
    """统一语义解析输出"""
    intent: Intent = Field(description="意图类型")
    dimensions: List[Dimension] = Field(default_factory=list, description="维度列表")
    measures: List[Measure] = Field(default_factory=list, description="度量列表")
    filters: List[Filter] = Field(default_factory=list, description="筛选条件")
    computation: Optional[Computation] = Field(default=None, description="复杂计算")
    clarification: Optional[str] = Field(default=None, description="澄清问题")
    reasoning: str = Field(description="推理过程")
```

```python
class ObserverDecision(str, Enum):
    """Observer 决策类型"""
    CONTINUE = "CONTINUE"  # 执行成功，继续下一步
    RETRY = "RETRY"        # 可恢复错误，重试指定步骤
    CLARIFY = "CLARIFY"    # 需要用户澄清

class RetryTarget(str, Enum):

    EXECUTE_QUERY = "execute_query"


class ObserverOutput(BaseModel):

    decision: ObserverDecision = Field(description="决策类型")

    retry_reason: Optional[str] = Field(

    retry_count: int = Field(default=0, description="当前重试次数")
```

## 7. 技术选型

### 7.1 核心框架

| 组件 | 技术选型 | 版本 | 说明 |
|------|----------|------|------|
| 编排框架 | LangGraph | 1.0.5+ | 核心编排框架（Subgraph 支持）|
| LLM 抽象 | LangChain | 1.1.3+ | LLM 抽象层 |
| 数据验证 | Pydantic | 2.x | 数据模型验证 |
| 向量存储 | FAISS | - | 高性能向量检索 |
| 异步支持 | aiohttp | 3.x | SSE 流式支持 |

### 7.2 中间件选型

| 中间件 | 来源 | 选型理由 |
|--------|------|---------|
| SummarizationMiddleware | LangChain | 内置实现，功能完善 |
| ModelRetryMiddleware | LangChain | 内置实现，支持指数退避 |
| ToolRetryMiddleware | LangChain | 内置实现，支持错误分类 |
| FilesystemMiddleware | 自定义 | 特定业务需求（大文件缓存）|

### 7.3 RAG 技术选型

| 组件 | 技术选型 | 选型理由 |
|------|----------|---------|
| 向量存储 | FAISS | 高性能、支持增量更新 |
| 检索策略 | 混合检索 | 向量检索 + BM25，提高召回率 |
| 重排序 | 可选 Reranker | 提高精度（可选）|
| 嵌入模型 | 自定义 | 通过 embeddings.py 管理 |

### 7.4 为什么不使用 LangChain RAG

| 方面 | LangChain RAG | 自定义实现 |
|------|--------------|-----------|
| 基本检索 | ✅ 支持 | ✅ 支持 |
| 置信度阈值 | ❌ 需要自定义 | ✅ 内置 |
| 高置信度快速路径 | ❌ 需要自定义 | ✅ 内置 |
| LLM Fallback | ❌ 需要自定义 | ✅ 内置 |
| 缓存管理 | ❌ 需要自定义 | ✅ 内置 |
| 增量更新 | ❌ 需要自定义 | ✅ 内置 |
| 延迟分解 | ❌ 需要自定义 | ✅ 内置 |

**结论**：字段映射是特定场景，需要精细控制，自定义实现更合适。

### 7.5 LLM 输出模式选型

| 模式 | 优点 | 缺点 | 选择 |
|------|------|------|------|
| with_structured_output | 类型安全、自动验证 | 不支持流式输出 | ❌ |
| JSON 模式 + Pydantic | 支持流式、灵活 | 需要手动解析 | ✅ |

**当前实现**：
```python
response = await call_llm_with_tools(
    llm=self.llm,
    prompt=prompt,
    tools=[],
    response_format={"type": "json_object"},  # JSON 模式
)
result = parse_json_response(response, SemanticQuery)  # Pydantic 验证
```
