# 附录 A：Prompt 设计模式

## 1. 各工具的特色模式

基于 Cursor、Windsurf、Devin AI、Claude Code 等 15+ 工具分析：

| 工具 | 特色模式 | 适用场景 |
|------|----------|----------|
| Cursor | XML 标签分块 + 自我纠错 | 复杂规则组织 |
| Windsurf | 记忆系统 + 计划管理 | 长对话上下文 |
| Devin AI | 双模式(Planning/Standard) + Think 工具 | 复杂任务分解 |
| Claude Code | 极简风格 + 专业客观 | 代码生成 |
| Cline | XML Schema 工具定义 + 步骤化执行 | 工具调用 |
| v0 | 领域专精 + 集成指南 | 特定框架 |
| Lovable | 设计系统优先 + 调试工具 | UI 开发 |
| Replit | 提案式响应 + 工具切换 | IDE 集成 |

## 2. XML 标签分块模式（来自 Cursor）

```xml
<identity>
你是 [角色名称]，[职责描述]。
</identity>

<capabilities>
你可以：
- 能力 1
- 能力 2

你不能：
- 限制 1
- 限制 2
</capabilities>

<decision_rules>
## 规则类别 A
<rule_a>
| 条件 | 结果 | 说明 |
|------|------|------|
| ... | ... | ... |
</rule_a>

## 规则类别 B
<rule_b>
IF 条件 THEN 动作
</rule_b>
</decision_rules>

<thinking_steps>
1. 第一步：...
2. 第二步：...
3. 第三步：...
</thinking_steps>

<examples>
## 正例
问题：...
分析：...
输出：...

## 反例
问题：...
错误输出：...
正确输出：...
原因：...
</examples>

<self_correction>
在输出前检查：
1. 检查项 1
2. 检查项 2
3. 检查项 3
</self_correction>
```

## 3. 双模式架构（来自 Devin AI / Cline）

```
┌─────────────────────────────────────────────────────────────────┐
│                      双模式架构                                  │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  Planning Mode (规划模式)                                       │
│  ├── 目标：收集信息，制定计划                                    │
│  ├── 行为：搜索、阅读、分析、提问                                │
│  ├── 输出：计划文档、澄清问题                                    │
│  └── 工具：搜索、读取、LSP、浏览器                               │
│                                                                 │
│  Standard/Execution Mode (执行模式)                             │
│  ├── 目标：执行计划，完成任务                                    │
│  ├── 行为：编辑、创建、运行、验证                                │
│  ├── 输出：代码、文件、命令结果                                  │
│  └── 工具：编辑、写入、Shell、部署                               │
│                                                                 │
│  模式切换：                                                      │
│  - Planning → Standard: 计划确认后                              │
│  - Standard → Planning: 遇到不确定情况时                         │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

## 4. Think 工具模式（来自 Devin AI）

```xml
<think>
必须使用 think 的场景：
1. 关键 Git/GitHub 决策前
2. 从探索代码转向修改代码时
3. 报告完成前的自我检查
4. 遇到意外困难时
5. 查看截图/图片时

think 的作用：
- 作为草稿本自由思考
- 用户看不到 think 内容
- 帮助做出更好的决策
</think>
```

## 5. 最佳实践总结

### 5.1 Prompt 设计

1. **身份定义简洁**：1-2 句话，明确角色和专长
2. **能力边界清晰**：明确列出能做和不能做的事
3. **规则用表格**：映射关系用表格，条件逻辑用 IF-THEN
4. **XML 标签分块**：复杂规则用 XML 标签分隔
5. **显式推理步骤**：要求按步骤思考
6. **正反例对比**：同时展示正确和错误示例
7. **自我检查清单**：输出前的验证规则

### 5.2 各模式速查表

| 模式 | 来源 | 核心要点 | 适用场景 |
|------|------|----------|----------|
| XML 标签分块 | Cursor | `<tag>内容</tag>` 分隔规则 | 复杂规则组织 |
| 双模式架构 | Devin/Cline | Planning + Execution 分离 | 复杂任务 |
| Think 工具 | Devin | 关键决策前显式推理 | 重要判断 |
| 记忆系统 | Windsurf | 主动保存/检索上下文 | 长对话 |
| 自我纠错 | Cursor | `<non_compliance>` 检查清单 | 质量保证 |
| 正反例对比 | v0 | 同时展示正确和错误 | 消歧义 |
| 领域专精 | v0 | 框架/集成专用规则 | 特定领域 |
| 设计系统优先 | Lovable | 先定义 tokens 再写组件 | UI 开发 |
| 提案式响应 | Replit | `<proposed_*>` 标签 | IDE 集成 |
| 极简风格 | Claude Code | 最少 token，直接回答 | 效率优先 |
