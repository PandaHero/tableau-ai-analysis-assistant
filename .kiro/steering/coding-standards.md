# Analytics Assistant 编码规范

## 1. 目的

本规范用于约束 Analytics Assistant 后端代码的日常开发、重构、测试和评审，目标是：

- 保持模块边界清晰，避免长期结构腐化
- 让配置、安全、日志、测试和接口演进有统一标准
- 让“新代码”和“修改过的旧代码”具备稳定可维护性

本规范是工程规范，不承载特定业务功能方案。某个功能的实现细节、流程约束、交互设计和算法步骤，应写入对应的 `spec` / `design` 文档，而不是写进全局编码规范。

## 2. 适用范围

本规范适用于以下后端 Python 代码：

- `analytics_assistant/src/`
- `analytics_assistant/tests/`
- `analytics_assistant/scripts/`
- 根目录下参与后端运行或测试的 Python 入口文件

不适用于：

- 前端代码
- 纯文档文件
- 与项目无关的临时本地实验文件

执行原则：

- 对新文件和当前修改过的文件严格执行
- 对未触及的历史代码，按重构和修复节奏逐步收敛，不做一次性全仓强制清洗

## 3. 规则等级

本规范中的规则分为三类：

- `必须`：默认强制执行，只有在明确记录例外理由时才能偏离
- `默认应当`：正常情况下应遵守，确有必要可偏离，但要在代码注释、变更说明或评审中说明原因
- `允许`：可选做法，按具体场景选择

如果出现例外，必须记录以下信息中的至少两项：

- 为什么不能按规范做
- 影响范围是什么
- 这是临时方案还是长期方案
- 何时或在什么条件下清理

## 4. 决策优先级

发生冲突时，按以下顺序判断：

1. 用户在当前任务中的明确要求
2. 已确认生效的 `spec` / `design` / API 合同
3. 本编码规范
4. 现有实现和历史习惯

补充规则：

- 文档不是“唯一标准”，而是当前标准之一
- 如果实现事实与文档不一致，应先判断哪一方仍然有效，再决定改代码还是改文档
- 如果某项改动会形成对外破坏性变更，必须先确认迁移方案

## 5. 目录结构与模块职责

### 5.1 顶层职责

- `agents/`：Agent、Prompt、节点和领域推理逻辑
- `core/`：跨模块共享的接口、异常、通用 Schema
- `infra/`：配置、存储、AI、RAG、种子数据等基础设施
- `orchestration/`：工作流编排、状态流转、图运行边界
- `platform/`：外部平台适配，如 Tableau
- `api/`：HTTP、SSE、鉴权、请求校验和接口投影

### 5.2 目录约束

- Prompt 必须放在模块自己的 `prompts/` 目录
- Pydantic 数据模型必须放在模块自己的 `schemas/` 目录，跨模块共享模型放在 `core/schemas/`
- 种子数据和规则模式必须放在 `infra/seeds/` 或模块自己的 `seeds/`，不要塞进 `app.yaml`
- 自定义异常必须放在 `core/exceptions.py` 或基础设施自己的异常模块中

### 5.3 不允许的内容

- 不要为了“看起来方便”在模块根目录随意堆放 Prompt、Schema、规则数据
- 不要在全局编码规范中写入某个特定功能的步骤型实现要求
- 不要为了兼容旧导入路径而创建纯转发文件，除非迁移方案明确要求

## 6. 依赖方向与导入规则

### 6.1 依赖方向

默认依赖方向如下：

- `core` 不能依赖 `infra`、`platform`、`agents`、`orchestration`、`api`
- `infra` 可以依赖 `core`
- `platform` 可以依赖 `core` 和 `infra`
- `agents` 可以依赖 `core`、`infra`、`platform`、`agents.base`
- `agents` 不应依赖 `api` 或 `orchestration`
- `orchestration` 可以依赖 `agents`、`core`、`infra`、`platform`
- `api` 可以依赖 `orchestration`、`core`、`infra`

如果确实需要突破这一方向，必须先解释为什么不能通过重构边界解决。

### 6.2 导入位置

- 默认必须在文件顶部导入
- 包内优先使用相对导入
- 跨包使用绝对导入
- 导入顺序为：标准库 -> 第三方 -> 项目跨包 -> 项目包内

### 6.3 延迟导入

默认禁止在函数或方法内部导入。

允许延迟导入的场景：

- 可选依赖仅在特定路径使用
- 插件式加载或启动时避免重型依赖副作用
- 获取全局单例会触发昂贵初始化，且顶部导入会造成明显问题

使用延迟导入时必须满足：

- 注释说明原因
- 不能用“避免循环依赖”作为唯一理由
- 优先先评估是否可以通过重构模块边界解决

## 7. 配置、环境变量与 `.env`

### 7.1 配置来源

运行时可调参数默认应放在 `analytics_assistant/config/app.yaml`，例如：

- 阈值
- 超时
- 重试次数
- 并发度
- 缓存 TTL
- 默认策略选择

### 7.2 `.env` 与敏感配置

项目当前已启用 `.env`。

规则如下：

- 敏感信息必须来自环境变量或 `.env`
- `app.yaml` 可以引用环境变量占位符
- `.env` 仅用于本地开发与测试，不得提交真实密钥
- 代码中不得硬编码 API Key、Token、密码、私钥

### 7.3 允许写在代码中的常量

以下内容不必强行配置化：

- 协议常量和稳定事件名
- Schema 版本号
- 固定 DDL 结构
- 默认回退值，如 `_DEFAULT_TIMEOUT`
- 测试常量和局部调试常量
- 不预期在部署时频繁调整的内部实现细节

原则是：只有“运行中可能需要调整”的东西才配置化。

### 7.4 配置读取

- 统一通过项目配置加载器读取配置
- 读取失败时允许使用代码中的 `_DEFAULT_*` 回退值
- 降级时必须记录 `warning` 日志

## 8. 数据模型与命名

### 8.1 模型放置

- 业务输入输出和中间结构默认使用 Pydantic，放在 `schemas/`
- 基础设施内部轻量数据结构允许使用 `dataclass`
- 同一概念只能有一个主模型定义，不要并行维护多个几乎相同的版本

### 8.2 共享模型

跨模块共享的数据模型应放在 `core/schemas/`，不要在多个模块里重复定义。

### 8.3 命名后缀约定

以下命名保持一致：

- `_id`：通用主键
- `_luid`：外部平台对象标识，如 Tableau datasource
- `_ref`：对磁盘对象、artifact、trace 或外部资源的引用
- `_path`：本地路径
- `_at`：时间点
- `_ms`：毫秒时长

### 8.4 归一化规则

- 用于匹配、检索、缓存键的字符串，应在存储和比较时使用一致的大小写规范
- 不要一处存大写、一处按小写比较

## 9. 实现要求

### 9.1 复用优先

默认应当优先复用：

- LangChain / LangGraph 已提供能力
- 项目内 `infra/`、`agents/base/`、`core/` 的已有设施

不应重复实现已有的存储、检索、流式包装、配置加载或通用校验逻辑。

### 9.2 禁止伪实现

禁止以下做法：

- 返回假的成功结果
- 用占位逻辑冒充完整功能
- 为了“先跑通”跳过关键校验却不显式标注

如果必须临时降级：

- 说明降级原因
- 暴露降级状态
- 记录日志
- 不得伪装成功能完整

### 9.3 拆分原则

当出现以下情况时，默认应当拆分：

- 单文件持续增长到难以理解或超过约 800 行
- 一个类承担三类以上明显不同职责
- 同一文件同时包含协议定义、业务逻辑、IO 编排和渲染投影
- 测试必须构造大量上下文才能覆盖一个函数

拆分优先方式：

- 抽服务函数
- 抽 `components/`
- 抽 `schemas/`
- 抽运行边界包装层

### 9.4 注释与文档

- 注释解释“为什么”，不要重复“做什么”
- 公开的非简单函数、类、模块应有简洁 docstring
- docstring 使用统一风格即可，推荐 Google 风格
- `TODO` / `FIXME` 必须写清原因和目标，禁止只写标签

## 10. 错误处理与日志

### 10.1 错误处理

- 优先捕获具体异常
- 捕获 `Exception` 时，必须记录足够上下文
- 重新抛出时应保留异常链
- 对外部依赖失败，要么包装成领域异常，要么显式降级

禁止：

- 空的 `except`
- 吞掉异常后静默返回
- 没有上下文的泛化错误日志

### 10.2 日志要求

- 统一使用 `logging.getLogger(__name__)`
- 禁止使用 `print`
- 关键日志应尽量带上可追踪字段，例如 `request_id`、`session_id`、`thread_id`、`run_id`
- 敏感信息必须脱敏

### 10.3 日志级别

- `debug`：内部状态、排查信息
- `info`：稳定业务节点和关键流程事件
- `warning`：可恢复降级、配置缺失、回退路径
- `error` / `exception`：失败或意外异常

## 11. 异步、并发与资源生命周期

### 11.1 异步规则

- `async` 函数中禁止阻塞 IO
- 独立异步操作应使用 `asyncio.gather`
- 对外部服务并发访问应加并发限制
- 外部调用必须设置超时

### 11.2 资源生命周期

- `httpx` client、数据库连接、文件句柄、流式迭代器必须正确关闭
- 流式输出和长任务应考虑取消、超时和异常清理
- 后台任务如果会跨请求存活，必须明确谁负责回收

### 11.3 流式与中断恢复

涉及 SSE、streaming、interrupt/resume 的代码应满足：

- 中断状态持久化边界清晰
- 普通用户视图与调试视图分离
- 出错时返回稳定错误对象，不把内部堆栈直接暴露给前端

## 12. 安全与边界控制

### 12.1 基本安全规则

- 生产默认开启 SSL 验证
- 禁止把敏感配置写死在源码或仓库内配置文件中
- 日志、异常、调试输出中禁止泄露完整密钥或认证头

### 12.2 多租户与缓存边界

缓存键、artifact 引用和持久化记录必须包含足够的作用域信息，至少覆盖：

- 用户或租户
- 数据源或业务对象
- 版本或 schema 边界（如果适用）

目标是避免跨租户、跨数据源串读和误命中。

### 12.3 文件访问边界

如果为模型或 API 暴露文件工作区：

- 默认只读
- 必须做路径 allowlist 校验
- 不允许越权访问工作区外文件

## 13. API、SSE 与公共合同

### 13.1 公共合同定义

以下视为公共合同：

- HTTP 请求与响应结构
- SSE 事件名和 payload 结构
- 持久化表结构对上层仓库的可见语义
- 对外部调用方承诺稳定的错误码

### 13.2 变更规则

修改公共合同时，默认应当：

- 明确这是兼容变更还是破坏性变更
- 补充迁移说明或版本策略
- 增加合同测试

禁止：

- 静默修改字段名或事件名
- 把内部节点名、调试 token、临时结构直接暴露成正式合同

## 14. 测试规范

### 14.1 测试分层

- `unit`：单元测试，验证纯逻辑、边界条件和异常分支
- `contract`：合同测试，验证 API、SSE、Schema、仓库接口等稳定结构
- `integration`：集成测试，验证真实模块拼装与本地基础设施交互
- `e2e` / `golden`：端到端验证，必要时连接真实外部服务
- `manual`：手动测试和排障脚本

### 14.2 Mock 策略

- 单元测试允许 mock 外部依赖
- 合同测试允许使用受控 stub / fake，重点验证结构稳定性
- 集成测试默认不 mock 仓库内部核心模块之间的真实组合
- 集成测试可以替换不稳定、昂贵或不可重复的第三方服务
- 真实外部服务调用应集中在少量 `e2e` / `golden` 测试中，不作为默认 CI 前提

### 14.3 测试要求

- 测试目录结构应尽量镜像源码结构
- 新增公共接口、事件、错误码时，应补对应合同测试
- 修复 bug 时，默认应补最小复现测试
- 测试名称应描述行为，而不是描述实现细节

## 15. 类型标注与代码风格

### 15.1 类型标注

- 公开函数和类方法默认应提供完整类型标注
- 非公开但复杂的函数也应尽量补类型
- 使用 Python 内置泛型，如 `list[str]`、`dict[str, Any]`
- 可空类型写法保持文件内一致，不要求全仓只允许一种语法

### 15.2 风格一致性

- 优先与所在文件、所在模块的既有风格一致
- 同一文件中不要混用多套风格
- 不要为微小风格偏好制造大规模无意义 diff

## 16. 兼容性与迁移

当前项目阶段默认不主动维护“无必要的兼容层”，但以下情况除外：

- 已形成外部 API 合同
- 已有前端或外部调用方依赖
- 正在执行中的迁移计划明确要求兼容期

如果要引入兼容层，必须说明：

- 为谁兼容
- 兼容多久
- 何时删除

## 17. 评审清单

提交前至少检查以下问题：

- 模块边界是否正确，是否出现反向依赖
- 是否把应进入 `app.yaml` 的参数写死了
- 是否把敏感信息放入源码、日志或测试快照
- 是否重复定义了已有模型或逻辑
- 是否引入了无说明的延迟导入
- 是否把公共合同做了静默变更
- 是否在错误处理里吞掉了上下文
- 是否为新增合同或 bug 修复补了测试

## 18. 例外记录模板

当必须偏离规范时，建议在代码注释或变更说明中使用以下格式：

```python
# 规范例外：
# 原因：这里需要延迟加载可选依赖，否则模块导入阶段会触发重型初始化。
# 范围：仅影响 Insight Agent 的可选图编译路径。
# 清理条件：当依赖注入边界稳定后，改回顶部导入。
```

## 19. 最终原则

- 写能维护的代码，不写只在今天能工作的代码
- 先保护边界，再谈局部技巧
- 先让合同稳定，再让内部实现演化
- 规范服务于工程效率，不服务于形式主义
