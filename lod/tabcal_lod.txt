我将为你系统地梳理关于LOD（详细级别表达式）‍ 和表计算的核心知识体系。这两者是现代商业智能与数据分析，特别是在Tableau等可视化工具中，实现复杂分析逻辑的基石。理解它们，意味着你能从“描述数据”跃升至“解释数据”。

第一部分：LOD（详细级别表达式）‍

核心定义：LOD是一种计算机制，它允许你独立于可视化视图的详细级别，定义一个固定的计算粒度。你可以把它理解为数据分析中的“锚点”——无论你的图表如何筛选、聚合或分组，这个计算都基于你预先设定的那个固定层级执行。

1. 三种基本类型

FIXED：{FIXED [维度] : 聚合表达式}

含义：完全无视视图中的筛选器（除非是上下文筛选器或数据源筛选器），严格在指定的维度组合上计算。

经典用例：计算每个客户的终身总消费额，然后在任何报表中（无论是按地区、按产品看）都能引用这个固定值，用于计算客户平均消费、识别高价值客户等。

示例：{FIXED [客户ID] : SUM([销售额])}

INCLUDE：{INCLUDE [维度] : 聚合表达式}

含义：添加更多细节到视图的当前详细级别中进行计算。视图的详细级别是“分母”，INCLUDE添加的维度是更细的“分子”。

核心逻辑：它受视图所有筛选器的影响，计算粒度比视图当前粒度更细。

经典用例：在视图只显示“地区”和“销售额”时，你想知道每个地区内“每个订单的平均销售额”。这时需要将“订单ID”这个更细的维度INCLUDE进来。

示例：视图粒度是[地区]，计算为 {INCLUDE [订单ID] : AVG([销售额])}，它会先按[地区], [订单ID]计算每个订单的销售额，然后在地区内求平均。

EXCLUDE：{EXCLUDE [维度] : 聚合表达式}

含义：移除视图中的某些维度细节，在更粗的粒度上计算。视图的详细级别是“分母”，EXCLUDE移除的维度是让计算变“粗”。

核心逻辑：它同样受视图筛选器影响，计算粒度比视图当前粒度更粗。

经典用例：视图显示了“产品子类别”的销售额，但你想同时看到该产品所属“大类”的销售额占比。这时需要EXCLUDE掉“子类别”维度，在大类层级计算总额。

示例：视图粒度是[子类别]，计算为 SUM([销售额]) / {EXCLUDE [子类别] : SUM([销售额])}，即可得到子类别占大类的百分比。

2. 核心区别与选择逻辑

FIXED vs INCLUDE/EXCLUDE：最根本的区别在于对视图筛选器的响应。FIXED是“固执的”，它自己定义世界；而INCLUDE/EXCLUDE是“灵活的”，它们在视图定义的世界里做加减法。

选择流程图：

你需要一个绝对不受视图布局影响的指标吗？ → 选 FIXED。

你需要一个受视图筛选影响，但计算粒度比视图更细的指标吗？ → 选 INCLUDE。

你需要一个受视图筛选影响，但计算粒度比视图更粗的指标吗？ → 选 EXCLUDE。

3. 高级应用与思维

嵌套与组合：LOD可以相互嵌套或与表计算组合，解决极其复杂的问题。例如，先用{FIXED [客户ID] : MAX([订单日期])}找到每个客户最近购买日期，再用表计算计算距离分析日期的天数。

集合与分组的替代方案：很多用“集合”或“分组”实现的功能（如识别复购客户），用LOD（{FIXED [客户ID] : COUNTD([订单日期])}）来实现更为动态和灵活。

性能考量：FIXED LOD通常在数据源查询时一次性计算完成，性能较好。而INCLUDE/EXCLUDE可能与视图渲染更紧密地结合，在超大数据集下需注意。

第二部分：表计算

核心定义：表计算是在数据检索结果（即“表”）之后进行的二次计算。它不向数据库发送新查询，而是在Tableau（或其他工具）内存中的数据结构上，沿着你指定的寻址和分区方向进行移动计算。

1. 两大核心概念

寻址：指计算的方向（“沿着什么算”）。例如“沿表向下”（Y轴）、“沿表交叉”（先横后纵）。这决定了计算的移动路径。

分区：指重置计算的范围（“在哪个范围内算”）。表计算会在每个分区内独立开始。例如，按“年份”分区，则每年的累计销售额会从零开始重新累计。

2. 常见函数与用途

排名与排序：RANK(), INDEX()

用于在分区内对值进行排序，如“每个地区内销售员的业绩排名”。

累计与移动计算：RUNNING_SUM(), WINDOW_SUM(), MOVING_AVG()

用于计算累计值（YTD累计）、移动平均值（MA）等。WINDOW_SUM()功能强大，可定义计算窗口（如前N行后N行）。

比例与差异：TOTAL(), DIFF(), PERCENTILE()

TOTAL()用于计算分区内总和，常用于计算“占比”。

DIFF()计算与前一行的差异，用于环比分析。

3. 配置方式

快速表计算：右键单击度量→“快速表计算”，选择“汇总”、“年同比增长”等预设模式。

自定义：通过“计算依据”菜单或拖拽“区”、“列”、“行”架子上的字段来精确定义分区和寻址。这是掌握表计算的关键。

4. 表计算 vs LOD：根本性哲学差异

这是理解两者最关键的部分。我们可以用一个比喻来区分：

LOD 像是定义数据的“本体”‍。它在查询时告诉数据库：“请按这个固定的真相来组织数据。” 结果是一个新的、静态的数据字段。

表计算 像是定义观察数据的“视角”或“透镜”‍。它在数据已经呈现为一张表后，告诉可视化工具：“请按照这种移动和比较的规则，重新解读这些已经存在的数据。” 结果是一个动态的、依赖于视图布局的计算值。

一个经典对比案例：

目标：计算“每个月的销售额占全年总额的百分比”。

用LOD（FIXED）实现：

SUM([销售额]) / {FIXED YEAR([订单日期]) : SUM([销售额])} 

逻辑：分子是每月销售额，分母是固定按年计算的销售额总和。这个百分比字段在添加到视图时就已经计算好了。

用表计算实现：

将YEAR([订单日期])拖到“颜色”或“详细信息”，将MONTH([订单日期])拖到列，SUM([销售额])拖到行。

右键单击行上的SUM([销售额])胶囊→“快速表计算”→“总额百分比”。

然后右键单击该胶囊→“计算依据”→“表（横穿）”。

逻辑：Tableau先得到每月销售额的列表，然后在这个列表上，从左到右（横穿）计算每个值占整个表（即全年）总和的比例。

关键区别：LOD方案中，分母{FIXED ...}是一个独立的、预计算的字段。表计算方案中，分母是视图渲染完成后，从整个数据切片中动态计算出来的总和。

第三部分：综合运用与思维进阶

在实际分析中，LOD和表计算往往联手解决复杂问题。

示例：分析每个客户本次购买与上次购买的时间间隔（购买周期）‍

用LOD定位事实：为每个客户找到其所有订单日期。

{FIXED [客户ID] : MAX([订单日期])} -> 最近购买日

要找到上一次购买日，需要更复杂的逻辑，可能涉及LOOKUP表计算或IF判断。

用表计算计算差值：在按客户ID和订单日期排序的列表中，使用DATEDIFF('day', LOOKUP(MIN([订单日期]), -1), MIN([订单日期]))。这里，LOOKUP(..., -1)是表计算，用于获取同一分区（客户ID）内上一行的日期。

给你的启发性问题： 当你面对一个分析需求时，可以问自己：

这个计算所需的“最细颗粒度事实”是什么？是否需要创造一个独立于当前视图的“锚点”？（这指向LOD）

这个计算是否本质上是一种“视角转换”，是对已经呈现在眼前的这组数字进行排序、比较、累计或比例换算？（这指向表计算）

我是否可以先利用LOD构建出基础的“原子指标”，然后再利用表计算对其进行灵活的“模式分析”？

掌握这两者，意味着你拥有了将原始数据转化为深刻洞察的“手术刀”和“显微镜”。它们让你不再受限于数据表的原始结构，而是能够自由地定义问题、构建指标，从而揭示数据背后真正的逻辑与故事。

还能再具体一些吗?最好给我一些具体的例子

好的，我们来深入探讨几个具体的例子，以阐明LOD表达式与表计算的区别、选择逻辑及组合应用。

例1：计算“每个细分的利润，占其所属类别总利润的百分比”‍

这是一个经典的“结构分析”问题。视图的详细级别是“细分”和“类别”的组合，但我们需要一个在“类别”层级上的聚合值作为分母。

方法A：使用表计算（WINDOW_SUM）‍

视图构建：将[类别]和[细分]放在行上，SUM([利润])放在列上，形成一个交叉表。

计算：对利润胶囊使用“快速表计算”->“合计百分比”。关键在于设置“计算依据”。

逻辑：我们需要在每个“类别”的分区内，计算其内部各“细分”的占比。因此，分区是[类别]（在每个类别内重置计算），寻址是[细分]（沿细分方向计算百分比）。最终计算为：SUM([利润]) / WINDOW_SUM(SUM([利润]))，并在每个类别分区内执行。

特点：计算完全依赖于视图的布局（行上的字段顺序）。如果改变视图结构，可能需要重新调整计算依据。

方法B：使用LOD表达式（EXCLUDE或FIXED）‍

计算字段：创建一个计算字段 类别利润总额。

使用EXCLUDE: {EXCLUDE [细分] : SUM([利润])}。意为：从当前视图详细级别中排除“细分”维度，计算聚合。

或使用FIXED: {FIXED [类别] : SUM([利润])}。意为：无视视图，固定按“类别”聚合利润。

最终计算：SUM([利润]) / [类别利润总额]。

区别：

EXCLUDE是相对指定，如果视图维度变化，其计算结果可能随之变化。

FIXED是绝对指定，计算结果与视图无关，仅由[类别]决定。并且，FIXED LOD的优先级高于视图中的维度筛选器，除非将筛选器置于上下文中。

如何选择：对于此类“在视图内计算更高层级聚合”的问题，表计算通常是更直接、性能更优的选择。但如果你希望该分母值成为一个固定的、可重用的字段，或在复杂筛选下保持稳定，FIXED LOD是更好的选择。

例2：识别并计算“新客户”数量（基于首次购买日期）‍

这个问题需要引入一个完全独立于当前视图的详细级别：每个客户。无论你的视图是分析时间趋势、产品类别还是地区，判断一个客户是否为新客户的标准，都基于其在整个数据集中的首次购买行为。

必须使用LOD表达式（FIXED）‍：

创建核心字段：客户首次购买日期 = {FIXED [客户ID] : MIN([订单日期])}。这个计算在“客户ID”的详细级别上执行，返回每个客户对应的最早订单日期。

逻辑判断：创建一个字段是否为新客户，例如：IF [订单日期] = [客户首次购买日期] THEN “新客户” ELSE “老客户” END。

进行分析：现在，你可以将[订单日期]（按年或按月）拖到列，将[是否为新客户]拖到颜色，并对[客户ID]进行计数（去重），即可看到每月或每年的新、老客户数量趋势。

为何表计算无法单独解决：表计算只能在已经聚合好的视图数据上进行运算。它无法回溯到原始数据行级别，去为每个客户寻找全局的首次购买日期。这正是LOD表达式的核心能力：在主视图详细级别之外，指定一个独立的详细级别进行聚合计算。

例3：高级组合——计算“每个客户的销售额，并分析其相对于所属区域最高消费客户的占比”‍

这个问题包含多个层次，需要LOD和表计算协同工作。

层次一（独立事实）‍：每个客户的销售额。 -> 使用FIXED LOD：{FIXED [客户ID] : SUM([销售额])}。

层次二（范围聚合）‍：每个区域内，所有客户的销售额。 -> 在包含[区域]和[客户ID]的视图中，可以使用表计算WINDOW_SUM(...)沿客户方向汇总；或使用EXCLUDE LOD：{EXCLUDE [客户ID] : SUM([销售额])}，从当前视图中排除客户维度。

层次三（范围极值）‍：每个区域内，最高消费客户的销售额。 -> 这需要在“区域”的详细级别上，对第一步得到的“每个客户销售额”求最大值。这引出了嵌套LOD表达式的概念：{EXCLUDE [客户ID] : MAX({FIXED [客户ID] : SUM([销售额])})}。内层FIXED计算每个客户的销售额，外层EXCLUDE在区域内求最大值。

最终计算：{FIXED [客户ID] : SUM([销售额])} / [区域最高客户销售额]。

这个例子展示了如何将LOD作为构建基础指标（客户销售额）的“原子”，然后通过更复杂的LOD嵌套或表计算，来构建高级分析模型（标杆对比分析）。

总结与选择框架

当你面对一个分析问题时，可以遵循以下思考路径：

明确计算所需的“详细级别”‍：我需要的这个数字，应该在哪个数据粒度上计算？（例如：每个订单行？每个客户？每个产品？每个地区？）

判断该级别与视图的关系：

与视图粒度相同：可能只需基本聚合或行级别计算。

比视图粒度更粗（更高聚合）‍：考虑表计算（如合计百分比、移动平均）或 EXCLUDE LOD。

比视图粒度更细（更低聚合）‍：考虑 INCLUDE LOD。

完全独立于视图：必须使用 FIXED LOD 来锚定这个事实。

考虑计算优先级与筛选器影响：如果计算需要规避视图中的维度筛选器，使用FIXED。如果计算需要遵从维度筛选器，使用INCLUDE、EXCLUDE或表计算。

性能与复杂度权衡：对于视图内的二次聚合，优先考虑表计算，因为它通常是本地计算，性能较好。对于复杂的、多层次的指标定义，则需借助LOD表达式。

本质上，LOD用于“定义数据”‍（创建新的数据颗粒度），而表计算用于“分析视图”‍（对已呈现的数据进行变换和解读）。掌握两者，你就能灵活地构建从基础指标到复杂洞察的完整分析链条。

例1：计算“每个产品子类别的销售额占其所属大类总销售额的百分比”‍

数据表结构示例（原始数据片段）：

订单ID产品大类产品子类别销售额001家具椅子100002家具桌子200003办公用品笔50004家具椅子150005办公用品纸80

目标视图（我们想看到的最终表格）：

产品大类产品子类别销售额占大类百分比家具椅子25055.6%家具桌子20044.4%办公用品笔5038.5%办公用品纸8061.5%

计算逻辑：

家具大类总销售额 = 250 + 200 = 450

椅子占家具百分比 = 250 / 450 = 55.6%

桌子占家具百分比 = 200 / 450 = 44.4%

办公用品大类总销售额 = 50 + 80 = 130

笔占办公用品百分比 = 50 / 130 = 38.5%

纸占办公用品百分比 = 80 / 130 = 61.5%

实现方法对比：

方法A：使用表计算（WINDOW_SUM）‍

构建视图：将[产品大类]和[产品子类别]拖到行，SUM([销售额])拖到值。

对销售额字段应用“快速表计算”->“合计百分比”。

关键设置：右键单击计算后的字段->“计算依据”->选择“产品大类”（分区）和“产品子类别”（寻址）。

底层计算过程：

在“家具”分区内：WINDOW_SUM(销售额) = 450

椅子行：250 / 450 = 55.6%

桌子行：200 / 450 = 44.4%

分区重置，进入“办公用品”分区：WINDOW_SUM(销售额) = 130

笔行：50 / 130 = 38.5%

纸行：80 / 130 = 61.5%

方法B：使用LOD表达式（FIXED）‍

创建计算字段大类总销售额：

{FIXED [产品大类] : SUM([销售额])} 

这个计算会为每一行原始数据都附加一个值：属于该产品大类的所有销售额总和。

附加后的逻辑视图：

订单ID产品大类产品子类别销售额大类总销售额001家具椅子100450002家具桌子200450003办公用品笔50130004家具椅子150450005办公用品纸80130

创建最终计算字段占大类百分比：

SUM([销售额]) / SUM([大类总销售额]) 

当按[产品大类, 产品子类别]聚合时，分子SUM([销售额])得到子类别销售额，分母SUM([大类总销售额])由于在每个大类内是常数，求和后仍为该常数。

例2：识别“高价值客户”（消费总额超过所在区域平均消费2倍）‍

数据表结构示例：

客户ID区域订单金额C001华东5000C002华东3000C003华东8000C004华北2000C005华北6000

分析目标： 找出每个区域内，个人总消费超过该区域客户平均消费2倍的客户。

分步计算逻辑：

步骤1：计算每个客户的总消费额（客户级别详细级别）‍

必须使用FIXED LOD，因为这是独立于区域分析的客户级别事实。

计算字段客户总消费：

{FIXED [客户ID] : SUM([订单金额])} 

附加后的逻辑视图：

客户ID区域订单金额客户总消费C001华东50005000C002华东30003000C003华东80008000C004华北20002000C005华北60006000

步骤2：计算每个区域的平均客户消费额（区域级别详细级别）‍

需要在区域级别上，对客户总消费求平均。

计算字段区域平均消费：

{FIXED [区域] : AVG([客户总消费])} 

或者更直接地：

{FIXED [区域] : AVG({FIXED [客户ID] : SUM([订单金额])})} 

这是一个嵌套LOD：内层计算每个客户的消费，外层在区域内求平均。

计算过程：

华东区域：客户消费为5000, 3000, 8000，平均值为(5000+3000+8000)/3 = 5333.33

华北区域：客户消费为2000, 6000，平均值为(2000+6000)/2 = 4000

步骤3：判断逻辑

计算字段是否高价值客户：

IF [客户总消费] > [区域平均消费] * 2  THEN "高价值客户"  ELSE "普通客户"  END 

最终判断结果：

客户ID区域客户总消费区域平均消费是否高价值客户C001华东50005333.33普通客户C002华东30005333.33普通客户C003华东80005333.33高价值客户C004华北20004000普通客户C005华北60004000高价值客户

解释： C003消费8000 > 5333.33*2=10666.67？等等，这里有个逻辑问题：8000并没有大于10666.67。这说明我们的阈值设定可能过于严格，或者示例数据需要调整。但重要的是你看到了计算逻辑的流程：先通过LOD构建基础指标（客户总消费），再通过LOD构建比较基准（区域平均消费），最后进行判断。

例3：计算“滚动三个月平均销售额”‍

这是一个典型的时间序列分析问题，非常适合用表计算解决。

数据表结构示例（月度聚合后）：

年月销售额2025-0110002025-0212002025-0311002025-0413002025-0514002025-061500

目标： 计算每个月的最近三个月移动平均值。

使用表计算（MOVING_AVG）的实现：

将[年月]拖到列（确保是连续日期或正确排序），SUM([销售额])拖到行。

右键单击销售额胶囊->“添加表计算”。

选择计算类型：“移动平均”。

设置参数：

前/后范围：选择“前”2个，“后”0个（表示用当前月及前两个月计算平均）

或直接设置“从起始到当前”的区间为3

计算过程推演：

2025-03的移动平均 = (2025-01:1000 + 2025-02:1200 + 2025-03:1100) / 3 = 1100

2025-04的移动平均 = (2025-02:1200 + 2025-03:1100 + 2025-04:1300) / 3 = 1200

2025-05的移动平均 = (2025-03:1100 + 2025-04:1300 + 2025-05:1400) / 3 = 1233.33

以此类推...

为什么这里用表计算而不用LOD？ 因为移动平均是典型的“视图相关”计算：它依赖于数据在视图中的顺序和范围。LOD表达式无法直接表达“当前行的前N行”这种相对位置概念，而表计算正是为此设计的。

思维实验：什么时候必须用LOD？

想象这个需求：“计算每个销售人员的业绩，并显示他完成了年度目标的百分之多少。”

这里有两个详细级别：

销售人员的实际销售额（可以从订单数据中{FIXED [销售人员ID] : SUM([销售额])}计算）

年度目标（通常存储在与销售人员维度关联的另一张目标表中，详细级别是[销售人员ID, 年度]）

要计算完成率，你需要将这两个不同详细级别的指标关联起来。LOD（特别是FIXED）能够将订单数据聚合到与目标表相同的详细级别（销售人员+年度），从而使得关联和计算成为可能。表计算无法跨这种异构的详细级别进行关联。

希望这些纯文本的推演过程能帮助你更透彻地理解LOD和表计算的内在逻辑。关键是要在脑海中构建出数据在不同详细级别之间流动和聚合的图景。复杂例1：会员RFM矩阵分析（客户分层的经典模型）‍

这是一个需要多步计算、构建多层指标体系的典型高级分析。

业务目标：将客户分为8个群体（2x2x2矩阵），基于以下三个维度：

R（Recency，近度）‍：客户最近一次购买距离分析日期的天数。

F（Frequency，频度）‍：客户在分析周期内的购买次数。

M（Monetary，价值）‍：客户在分析周期内的消费总额。

数据与计算推演：

假设分析日期为 2025-12-22，分析周期为过去一年（2024-12-22 至 2025-12-22）。

步骤1：计算每个客户的R、F、M基础值 这需要在客户的详细级别上进行计算，完全独立于任何后续的视图。必须使用FIXED LOD。

客户最近购买日 = {FIXED [客户ID] : MAX([订单日期])}

客户近度R = DATEDIFF('day', [客户最近购买日], DATE('2025-12-22'))

客户频度F = {FIXED [客户ID] : COUNTD([订单ID])} （统计唯一订单数）

客户价值M = {FIXED [客户ID] : SUM([销售额])}

步骤2：确定R、F、M的阈值，并对客户进行分层 通常使用所有客户的中位数或平均值作为阈值。这里需要计算全局的聚合值作为标尺。

近度阈值 = {FIXED : MEDIAN([客户近度R])} （注意：{FIXED : ...} 表示在整个数据源层面聚合）

频度阈值 = {FIXED : MEDIAN([客户频度F])}

价值阈值 = {FIXED : MEDIAN([客户价值M])}

接着，为每个客户打上高低标签：

R等级 = IF [客户近度R] <= [近度阈值] THEN “高” ELSE “低” END （近度值越小越好）

F等级 = IF [客户频度F] >= [频度阈值] THEN “高” ELSE “低” END

M等级 = IF [客户价值M] >= [价值阈值] THEN “高” ELSE “低” END

RFM分组 = [R等级] + [F等级] + [M等级] （例如“高高高”、“高低低”）

步骤3：在视图中进行聚合分析 现在，可以将[RFM分组]作为维度，进行各种聚合分析，例如：

计算每个RFM分组的客户数量、总销售额、平均利润。

使用表计算分析趋势，例如计算每个RFM分组销售额的环比增长：这需要在时间序列视图上，对每个分组内的销售额聚合进行行间差异计算。

要计算“每个RFM分组客户数占总客户数的百分比”，可以使用EXCLUDE LOD或表计算。例如，使用表计算SUM([客户数]) / TOTAL(SUM([客户数]))，并将“计算依据”设置为[RFM分组]，这实现了在视图详细级别上对更高聚合层次（总计）的引用。

核心挑战：这个模型完美展示了FIXED LOD如何用于构建分析维度（客户级别的RFM标签），而后续的分析则在这个新维度上，灵活运用聚合计算和表计算来揭示业务洞察。高级互动的关键在于把复杂问题拆解，基于问题焦点构建主视图，并选择合适的图形增加深度。

复杂例2：帕累托分析（二八定律可视化）‍

业务目标：找出贡献了80%销售额的那20%的产品是哪些，并动态展示。

计算推演：

步骤1：按产品计算累计销售额与累计百分比

视图：将[产品名称]按销售额降序排列，SUM([销售额])作为条形图。

创建第一个表计算字段 累计销售额：

RUNNING_SUM(SUM([销售额])) 

此计算沿着产品排列的顺序（寻址），从第一个产品开始累加销售额。

创建第二个表计算字段 累计百分比：

[累计销售额] / TOTAL(SUM([销售额])) 

或者直接用快速表计算“合计百分比”。这里TOTAL()计算了所有产品的销售总额，作为分母。

步骤2：动态参数与集动作结合

创建一个参数[目标百分比]，允许用户输入0到100之间的值（例如80）。

创建一个计算字段是否核心产品：

[累计百分比] <= [目标百分比]/100 

基于此计算字段创建一个集，命名为[核心产品集]。

将[核心产品集]拖到颜色上，即可动态高亮显示贡献了指定百分比销售额的产品群体。

核心挑战：此例的核心是表计算的顺序和方向。RUNNING_SUM的计算必须基于已排序的[产品名称]，且“计算依据”需设置为沿[产品名称]（寻址）。任何视图层次的变化都可能打断这个计算顺序。这体现了表计算的独特性：它是在视图级别上对聚合数据做二次计算，是基于维度的行间计算。同时，它结合了参数和集动作，实现了高级互动，让分析从静态报告变为动态探索工具。

复杂例3：同期群分析（Cohort Analysis）与复购率计算

业务目标：分析不同时期获取的客户群（例如每月新增客户），在后续各时间段的留存与复购情况。

计算推演：

步骤1：定义“客户群”与“分析周期”‍ 这需要两个独立的详细级别，并通过LOD将它们关联起来。

客户群维度（Cohort）‍：使用FIXED LOD确定每个客户的归属时间。

客户首购月份 = {FIXED [客户ID] : MIN(DATETRUNC(‘month’, [订单日期]))}

此字段将作为“客户群”标签，例如“2025-01新客”。

分析周期维度（Period）‍：通常使用订单日期所在的月份。

订单月份 = DATETRUNC(‘month’, [订单日期])

计算时间间隔：

间隔期数 = DATEDIFF(‘month’, [客户首购月份], [订单月份])

结果0表示首购月，1表示次月，以此类推。

步骤2：构建矩阵并计算复购客户数

视图：将[客户首购月份]放在行，[间隔期数]放在列。

度量：将[客户ID]拖到文本，并设置为“计数(不同)”，得到每个客户群在不同间隔期内的去重客户数。

此时视图是一个标准的“客户数”矩阵。

步骤3：计算复购率（需要高级计算转换）‍ 复购率通常表示为：第N月的留存客户数 / 首月客户数。

我们需要一个计算字段，来获取每个客户群在“间隔期数=0”时的基准客户数。这需要引用独立于当前视图单元格的另一个聚合值。这里就是LOD表达式的关键应用场景。

创建计算字段首月客户数：

{FIXED [客户首购月份] : COUNTD([客户ID])} 

注意：在包含[间隔期数]的视图中直接使用此字段，它会返回该客户群在所有月份的总客户数，而非首月数。因此需要结合筛选逻辑。

更精确的做法是使用包含条件判断的LOD，或利用表计算的LOOKUP函数来获取首月值。但最清晰的逻辑是：先创建一个仅包含首月数据的FIXED LOD计算，再在视图中与其他数据结合。这体现了在复杂模型中，多种计算需要分阶段、嵌套使用。

核心挑战：同期群分析的本质是一个视图，多个详细级别。主视图的详细级别是[客户首购月份， 订单月份]，但分析中不断需要引用[客户ID]级别的聚合（确定客户归属），以及[客户首购月份]级别的聚合（作为计算基准）。这完全符合LOD表达式被设计来解决的问题：回答有关单一视图中包含多个数据颗粒度级别的问题。选择正确的计算类型，取决于你需要引用的额外层次与主视图层次的关系是更高、更低还是独立。

这些复杂例子表明，真正的分析能力在于将业务问题解构为层次分明的数据问题，然后像搭积木一样，组合使用LOD定义“数据事实”，用表计算构建“分析视角”，最终通过可视化呈现洞察。
